## 🛠️ 技术决策调整

### 架构设计调整
1. **从模拟数据到真实数据库**
   - 原计划：使用模拟数据，逐步替换为API
   - 新方案：直接创建真实数据库表，完全移除模拟数据
   - 优势：避免技术债务，建立可持续的架构

2. **从资产表到业务表**
   - 原计划：通用资产表存储所有资源
   - 新方案：专用业务表（用户、文章、公告）存储数据和资产位置
   - 优势：更好的类型安全，更清晰的业务逻辑

3. **联邦身份集成**
   - 新增：支持Fediverse用户身份格式（@user@host）
   - 优势：为未来联邦登录和跨平台集成奠定基础

### 开发流程调整
1. **质量保证前置**
   - 原计划：开发完成后进行测试
   - 新方案：测试与开发并行，TDD关键路径
   - 优势：早期发现问题，减少返工

2. **持续集成强化**
   - 原计划：第三阶段建立CI/CD
   - 新方案：第二阶段即建立基础CI流程
   - 优势：确保代码质量持续可控

### 技术栈微调
1. **测试框架选择**
   - 前端：Jest + React Testing Library
   - 后端：Jest + Supertest
   - E2E：Playwright（可选）

2. **真实数据策略**
   - 开发环境：直接使用真实数据库和API
   - 测试环境：使用测试数据库和Jest mock
   - 优势：确保开发环境与生产环境一致，减少集成问题
